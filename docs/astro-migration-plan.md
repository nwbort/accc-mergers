# Migration Plan: React SPA → Astro Static Site

## Phase 0: Project Setup

### 0.1 — Initialize Astro project alongside the existing React app
- Create a new `merger-tracker/astro/` directory (sibling to `frontend/`)
- `npm create astro@latest` with the "empty" template
- Add integrations: `@astrojs/tailwind`, `@astrojs/cloudflare` (for adapter if needed later, though pure static is fine)
- Copy over `tailwind.config.js` and `postcss.config.js` from `frontend/`
- Copy the `public/` directory (favicons, manifest, robots.txt, sitemap, feed.xml)
- **Do NOT copy `public/data/`** — instead symlink it or configure Astro to reference it, since these files are generated by the Python pipeline

### 0.2 — Configure the data pipeline
- Astro can import JSON at build time. Create a `src/data/` loader module that reads the JSON files from the data directory
- Example: `const stats = await import('../../public/data/stats.json')` or use `Astro.glob()` / `fs.readFileSync` in `.astro` frontmatter
- This replaces all the `fetch()` calls that currently happen client-side — the data is now available at build time

### 0.3 — Update `generate_static_data.py` output path
- Change output from `merger-tracker/frontend/public/data/` to `merger-tracker/astro/public/data/` (or make it configurable)
- Alternatively, keep the output path the same and just move the frontend directory later

> **Pitfall:** Astro's content collections expect files in `src/content/`. You don't need to use content collections — plain JSON imports are fine and simpler for your data shape. Don't over-engineer this.

## Phase 1: Layout & Static Shell

### 1.1 — Create the base layout (`src/layouts/Base.astro`)
- Port the `<head>` from `index.html` (meta tags, OG tags, Google Fonts, favicon links)
- Replace `react-helmet-async` with Astro's native `<head>` — each page sets its own title/description via props to the layout
- Include the Tailwind CSS import
- Include `<slot />` for page content

### 1.2 — Create the Navbar component (`src/components/Navbar.astro`)
- Port the HTML structure from `Navbar.jsx`
- Navigation links become `<a href="...">` — Astro does file-based routing, no client-side router needed
- Active link highlighting: use `Astro.url.pathname` in the frontmatter to determine which link is active
- Mobile menu toggle: small vanilla JS `<script>` tag (10 lines)
- Scroll hide/show: small vanilla JS `<script>` tag (15 lines)
- **Defer** the notification bell/panel to Phase 4 (tracking island)

### 1.3 — Create the Footer component (`src/components/Footer.astro`)
- Direct port — it's already fully static

### 1.4 — Port `index.css`
- Copy as-is into `src/styles/global.css` and import in the base layout

> **Pitfall:** Astro components use `.astro` syntax, not JSX. The syntax is similar but different — no `className` (use `class`), no `{}` expressions in attributes (use `{expression}`), no fragments. Plan for a manual conversion pass, not a find-and-replace.

## Phase 2: Static Pages (easiest wins first)

### 2.1 — NickTwort page → `src/pages/nick-twort.astro`
- Fully static, no data fetching. Straight HTML port.
- Test that the route works at `/nick-twort`

### 2.2 — NotFound page → `src/pages/404.astro`
- Astro automatically serves `404.astro` for missing routes

### 2.3 — Commentary page → `src/pages/commentary.astro`
- Read `commentary.json` in frontmatter
- Render markdown content using Astro's built-in `<Fragment set:html={...} />` or install `marked` for markdown → HTML conversion at build time (replaces `react-markdown` which was client-side)
- StatusBadge, WaiverBadge, ExternalLinkIcon become `.astro` components — trivial port since they're pure presentational

### 2.4 — Timeline page → `src/pages/timeline.astro`
- Read all timeline page JSON files in frontmatter, concatenate, reverse (newest first)
- Render ALL events as static HTML — no infinite scroll needed since it's pre-rendered
- If the page gets too long (it won't for ~500 mergers), add CSS-only pagination or a "show more" with vanilla JS
- Port the color-coded event indicators as static CSS classes

### 2.5 — Industries page → `src/pages/industries.astro`
- Read `industries.json` in frontmatter
- Each industry row uses `<details><summary>` for expand/collapse — **zero JS**, native HTML
- For each industry, read the industry detail JSON and render the merger list inside the `<details>` element at build time (no lazy loading needed — the data is tiny)

### 2.6 — IndustryDetail pages → `src/pages/industries/[code].astro`
- Astro dynamic routes: `getStaticPaths()` returns all industry codes from `industries.json`
- Each page reads its industry JSON and renders statically

### 2.7 — Digest page → `src/pages/digest.astro`
- Read `digest.json` in frontmatter
- Port all the color-coded sections (DigestSection, MergerNameCell, DeterminationCell) as Astro components
- Markdown rendering: convert at build time with `marked` instead of `react-markdown`
- Summary card click-to-scroll: `<a href="#section-id">` with `scroll-behavior: smooth` in CSS
- **Email signup form**: keep as a small vanilla JS `<script>` at the bottom of the page (Turnstile + fetch POST). No framework needed — it's a single form with ~40 lines of JS

> **Pitfall:** The Tailwind safelist issue. `Digest.jsx` uses `COLOR_CLASSES` to avoid Tailwind purging dynamic class names. In Astro, the same approach works — keep the COLOR_CLASSES object and reference full class names. Alternatively, since you're rendering at build time, Tailwind's scanner will see the classes in the `.astro` template and include them automatically.

## Phase 3: Data-Heavy Pages

### 3.1 — Dashboard page → `src/pages/index.astro`
- Read `stats.json` and `upcoming-events.json` in frontmatter
- StatCards: pure HTML, link to `/mergers?status=...` (these links still work — the mergers page will handle the query params client-side)
- Recent mergers / recent determinations tables: static HTML
- Phase 1 duration progress bars: static HTML with `style="width: {percentage}%"`
- **Charts (Doughnut)**: Two options:
  - **Option A (recommended):** Pre-render as SVG at build time. Write a small helper that generates SVG doughnut charts from the data. Zero JS shipped.
  - **Option B:** Keep Chart.js as a client-side script. Add a `<canvas>` element and a `<script>` that initializes Chart.js on load. ~50KB of JS but simpler to implement.
- "New" badge (lastVisit): This relies on localStorage. Either drop it (minor feature) or implement as a small vanilla JS script that checks localStorage and toggles visibility of `.new-badge` elements

### 3.2 — Merger detail pages → `src/pages/mergers/[id].astro`
- `getStaticPaths()` returns all merger IDs from the list-page JSON files (or read `mergers.json` directly)
- Each page reads its individual merger JSON and renders statically
- Party expandables: `<details><summary>` — zero JS
- Markdown description/commentary: render at build time
- Event timeline: static HTML
- Track button: deferred to Phase 4 (tracking island)
- **This generates one HTML file per merger** — perfect for SEO, each page is its own URL

### 3.3 — Mergers list page → `src/pages/mergers.astro` (the hardest page)

This is the most complex page because of search, filtering, sorting, keyboard navigation, and infinite scroll. Strategy:

- **Render ALL mergers as static HTML** in the page (hidden by default with a `data-*` attribute system or just visible)
- **Search**: Use [Pagefind](https://pagefind.app/) — it's built for Astro, runs at build time to create a search index, and provides a tiny (~6KB) client-side search UI. It indexes your static HTML pages automatically. This replaces the custom `searchIndex.js` with zero custom code.
- **Filtering (phase, status)**: Two approaches:
  - **Option A (simpler):** Vanilla JS (~60 lines) that reads URL params and toggles `hidden` class on merger cards based on `data-phase`, `data-status`, `data-waiver` attributes baked into the HTML
  - **Option B:** Pre-generate filtered pages at build time (`/mergers/phase-1`, `/mergers/approved`, etc.) — but this doesn't compose well with combined filters
  - **Recommend Option A** — it's the same mental model as the current React version but much simpler
- **Sorting**: Vanilla JS that re-orders DOM elements based on `data-notification-date` and `data-determination-date` attributes. ~30 lines.
- **Infinite scroll / pagination**: Since all HTML is already on the page, use CSS `content-visibility: auto` for rendering performance, or a vanilla JS "show more" button that unhides the next batch. Much simpler than the current IntersectionObserver + fetch approach.
- **Keyboard navigation (j/k/Enter)**: Port `useKeyboardShortcuts.js` to vanilla JS. ~50 lines.

### 3.4 — Analysis page → `src/pages/analysis.astro`
- Stat cards: static HTML
- **Charts (Scatter, Bar)**: These are interactive (click to navigate, hover tooltips). Keep Chart.js as a client-side script.
  - Render `<canvas>` elements in the Astro template
  - Add a `<script>` block that imports Chart.js and initializes the charts with data embedded as a JSON `<script type="application/json">` block
  - Click handler navigates with `window.location.href`
  - This is ~80 lines of vanilla JS + Chart.js
  - **Alternative:** Use a lighter charting library like uPlot (~35KB vs Chart.js ~200KB)

> **Pitfall:** Chart.js tree-shaking. In the React version you manually register components (`ChartJS.register(...)`). In vanilla JS you'll need to do the same or use the auto-registration. Make sure you only import what you need.

## Phase 4: Interactive Islands (Tracking System)

This is the one feature that genuinely needs client-side state management. Two options:

### Option A — Vanilla JS island (recommended)
- Create `src/scripts/tracking.js` (~200 lines)
- Reads/writes localStorage (same keys as current: `merger_tracker_tracked`, `merger_tracker_seen_events`)
- On page load, checks which mergers are tracked and updates bell icons
- Notification panel: rendered as hidden HTML in the Navbar, JS toggles visibility and populates content
- Track/untrack buttons: JS event listeners on `[data-track-merger]` buttons
- Event fetching: fetch individual merger JSONs for tracked mergers (same as current, but vanilla fetch)
- This is self-contained and doesn't need any framework

### Option B — Preact island (if Option A feels too complex)
- Install `@astrojs/preact` (Preact is 3KB vs React's 40KB)
- Keep `TrackingContext.jsx` and `NotificationPanel.jsx` largely as-is, converting from React to Preact (almost no changes needed)
- Wrap them in `client:load` directive in Astro templates
- This adds ~3KB of JS but preserves the existing component logic

> **Pitfall:** The tracking system currently fetches individual merger detail JSONs on the client to get events. This still works in Astro since those JSON files are in `public/data/mergers/{id}.json`. No change needed to the data pipeline.

> **Pitfall:** The notification panel currently uses React Router's `<Link>` for navigation. In vanilla JS, use plain `<a href="...">` tags instead. In Preact island, use `<a>` tags (no router needed for simple links).

## Key Pitfalls to Watch

1. **Astro `.astro` syntax vs JSX** — `class` not `className`, different expression syntax. Plan for a manual conversion pass, not a find-and-replace.
2. **`getStaticPaths()` for dynamic routes** — must return every possible parameter value. Read from your JSON data files.
3. **Tailwind purging** — make sure all dynamic class names (especially the digest color classes) are visible to Tailwind's scanner at build time.
4. **Chart.js in vanilla JS** — slightly different initialization than react-chartjs-2. Test the click handlers carefully.
5. **Pagefind indexing** — runs after build, so you need to add a post-build step. Astro has an official integration for this.
6. **`mergers.json` is 443KB** — building ~120+ individual merger pages from `getStaticPaths` is fine, but make sure the build step reads the paginated list files, not the full 443KB file, to keep memory usage reasonable.
7. **Email signup Turnstile** — the Cloudflare Turnstile widget is framework-agnostic and works with vanilla JS. Don't overcomplicate it.
8. **The `matters/` directory copy** — the build command currently copies `../../matters` into `dist/matters`. Make sure this still works with the new directory structure.
9. **Filter state in URL** — the current `/mergers?status=X&phase=Y` pattern needs the vanilla JS filter script to read `URLSearchParams` on page load and apply filters. Test shared/bookmarked filter URLs.
10. **Session storage for filter persistence** — currently stores `mergers_filter_params` in sessionStorage so filters persist across navigation. Vanilla JS can do the same, but since Astro does full page navigations (not SPA), you'll want to use `sessionStorage` to restore filter state on page load. Alternatively, use Astro's `ViewTransitions` for SPA-like navigation without a framework.

## Bonus: View Transitions

Astro supports [View Transitions](https://docs.astro.build/en/guides/view-transitions/) — a browser API that gives you SPA-like page transitions without any JS framework. Adding `<ViewTransitions />` to your base layout means:
- Page navigations feel instant (no full reload flash)
- Shared elements (navbar) persist across pages
- Browser back/forward works naturally
- Filter state in URLs just works

This is optional but would make the site feel as smooth as the current React SPA, with zero JS cost. Worth enabling in Phase 1.
